
# EpicBook Infrastructure and Deployment Guide

This repository provisions AWS infrastructure with Terraform and deploys/configures the EpicBook application with Ansible.

## 1. What This Project Does

- Terraform creates:
  - Networking (VPC, subnets, route table, internet gateway)
  - Security groups (EC2 SSH/HTTP and RDS MySQL)
  - EC2 instance for the app
  - RDS MySQL instance for the database
  - Elastic IP for stable public access
  - Ansible inventory file generated automatically from Terraform outputs
- Ansible configures:
  - Base OS packages and SSH hardening
  - Nginx reverse proxy
  - App checkout and runtime setup (Node.js, PM2)
  - Database schema and seed import

## 2. Repository Structure (Terraform -> Ansible)

```text
.
|-- .gitignore
|-- README.md
|-- terraform/
|   `-- aws/
|       |-- main.tf
|       |-- providers.tf
|       |-- variables.tf
|       |-- outputs.tf
|       `-- terraform.tfvars            # local secrets/values (do not commit)
`-- ansible/
    |-- ansible.cfg
    |-- site.yml
    |-- inventory.ini                   # generated by Terraform (ignored)
    |-- inventory.ini.example
    |-- .vault_pass                     # local vault password file (ignored)
    |-- group_vars/
    |   `-- web/
    |       |-- main.yml                # non-secret vars
    |       `-- vault.yml               # encrypted secrets (or local secret vars)
    `-- roles/
        |-- common/
        |   |-- tasks/main.yml
        |   `-- handlers/main.yml
        |-- nginx/
        |   |-- tasks/main.yml
        |   |-- handlers/main.yml
        |   `-- templates/epicbook.conf.j2
        `-- epicbook/
            |-- tasks/main.yml
            `-- handlers/main.yml
```

### Terraform file responsibilities

- `terraform/aws/providers.tf`
  - Terraform and AWS provider version constraints.
  - AWS provider uses `var.region`.

- `terraform/aws/variables.tf`
  - Input contract for Terraform.
  - Defines typed variables with descriptions.

- `terraform/aws/terraform.tfvars`
  - Local concrete values for variables.
  - Holds sensitive values like `db_password`.
  - Should remain local and not be committed.

- `terraform/aws/main.tf`
  - Core infrastructure resources.
  - Creates an AWS key pair from local public key path.
  - Writes `ansible/inventory.ini` via `local_file` using created EC2 IP and DB host.

- `terraform/aws/outputs.tf`
  - Exposes selected outputs (for example public IP and DB host).

### Ansible file responsibilities

- `ansible/ansible.cfg`
  - Defaults for inventory, vault password file, SSH behavior.

- `ansible/site.yml`
  - Orchestration playbook. Applies roles in order: `common` -> `nginx` -> `epicbook`.

- `ansible/group_vars/web/main.yml`
  - Non-sensitive app/deploy configuration.
  - Must align DB user/database name with Terraform RDS settings.

- `ansible/group_vars/web/vault.yml`
  - Sensitive Ansible variables (recommended encrypted with ansible-vault).
  - Commonly includes `db_password` and other secrets.

- `ansible/inventory.ini`
  - Runtime inventory used by Ansible.
  - Auto-generated by Terraform. Do not hand-edit long term.

## 3. Important Things To Keep In Mind

### 3.1 Variable alignment between Terraform and Ansible

Keep these values consistent between Terraform and Ansible:

- `db_user`
- `db_name`
- `db_password`

If Terraform creates RDS with one DB username (for example `admin`) and Ansible tries another one, schema import fails with MySQL access denied.

### 3.2 Inventory is generated by Terraform

`ansible/inventory.ini` is generated from Terraform and may change whenever EC2 public IP changes.
Always use the current generated IP.

### 3.3 SSH key path must exist locally

`terraform/aws/main.tf` creates AWS key pair from:

- `ssh_public_key_path` in `terraform/aws/terraform.tfvars`

This must point to an existing local `.pub` file. Terraform will fail if the file is missing.

### 3.4 Idempotency expectations

Ansible tasks are adjusted so repeated runs should be mostly `ok` when nothing changed.
A normal stable run should show `changed=0` (or close to 0 if actual drift happened).

### 3.5 Security notes

- Do not commit secrets.
- Keep `terraform.tfvars`, `ansible/.vault_pass`, and plaintext secret files local.
- Prefer `ansible-vault` for `ansible/group_vars/web/vault.yml`.

## 4. Prerequisites

Install and configure:

- AWS account and credentials (for example via `aws configure`)
- Terraform (>= 1.3)
- Ansible
- SSH key pair on your machine (public key available as `.pub`)
- Python 3 environment if you use local virtualenv workflow

Optional but useful:

- `mysql-client` locally for manual DB checks

## 5. Credentials and Ignored Files (Examples)

The following files are sensitive/local and should not be committed.
Examples below are safe templates.

### 5.1 Example `terraform/aws/terraform.tfvars`

```hcl
region              = "us-east-2"
resource_prefix     = "epicbook"
instance_type       = "t3.micro"
admin_username      = "ubuntu"
ssh_public_key_path = "~/.ssh/id_rsa_epicbook.pub"

db_name             = "bookstore"
db_user             = "admin"
db_password         = "REPLACE_WITH_STRONG_PASSWORD"
```

### 5.2 Example `ansible/group_vars/web/main.yml`

```yaml
---
app_repo: "https://github.com/pravinmishraaws/theepicbook.git"
app_dest: "/var/www/epicbook"
app_user: "www-data"
app_group: "www-data"
nginx_site_name: "epicbook"
app_port: 3000

db_user: "admin"
db_name: "bookstore"
```

### 5.3 Example `ansible/group_vars/web/vault.yml` (plaintext template)

Use this only as a template, then encrypt it.

```yaml
---
db_password: "REPLACE_WITH_STRONG_PASSWORD"
```

Encrypt it:

```bash
cd ansible
ansible-vault encrypt group_vars/web/vault.yml
```

### 5.4 Example `ansible/.vault_pass`

```text
REPLACE_WITH_LOCAL_VAULT_PASSWORD
```

Set file permissions:

```bash
chmod 600 ansible/.vault_pass
```

### 5.5 Example `ansible/inventory.ini.example`

```ini
[web]
<EC2_PUBLIC_IP>

[web:vars]
ansible_user=ubuntu
ansible_ssh_private_key_file=/home/your-user/.ssh/id_rsa_epicbook
ansible_python_interpreter=/usr/bin/python3
db_host=<RDS_ENDPOINT>
```

## 6. Step-by-Step Deployment Guide

## Step 1: Clone repository

```bash
git clone <your-repo-url>
cd epicbook-ansible-terraform
```

## Step 2: Create SSH key pair (if you do not already have one)

Terraform reads your public key from `ssh_public_key_path`, so create a key pair first.

```bash
mkdir -p ~/.ssh
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa_epicbook -N ""
chmod 600 ~/.ssh/id_rsa_epicbook
chmod 644 ~/.ssh/id_rsa_epicbook.pub
```

Verify key files exist:

```bash
ls -l ~/.ssh/id_rsa_epicbook ~/.ssh/id_rsa_epicbook.pub
```

## Step 3: Prepare Terraform variable values

Create/update local tfvars:

```bash
cd terraform/aws
cp terraform.tfvars terraform.tfvars.local.backup 2>/dev/null || true
```

Edit `terraform.tfvars` with your real values and valid SSH public key path.

## Step 4: Initialize and validate Terraform

```bash
cd terraform/aws
terraform init
terraform validate
```

## Step 5: Plan infrastructure

```bash
cd terraform/aws
terraform plan
```

Check carefully:

- Region is correct.
- EC2 and RDS names are expected.
- DB username/password match what Ansible will use.

## Step 6: Apply infrastructure

```bash
cd terraform/aws
terraform apply
```

After apply, verify:

- `ansible/inventory.ini` was generated.
- Public IP in inventory looks correct.
- `db_host` is populated with RDS endpoint.

## Step 7: Prepare Ansible secrets

Create/verify local secret files.

- `ansible/.vault_pass`
- `ansible/group_vars/web/vault.yml` (prefer encrypted)

If using encrypted vault file:

```bash
cd ansible
ansible-vault view group_vars/web/vault.yml
```

## Step 8: Verify SSH access to EC2

Use the host from generated inventory.

```bash
ssh -i ~/.ssh/id_rsa_epicbook ubuntu@<EC2_PUBLIC_IP> hostname
```

If this fails:

- Check you are using the current IP from `ansible/inventory.ini`.
- Confirm EC2 security group allows TCP/22 from your source.
- Confirm private key matches `ssh_public_key_path` used by Terraform.

## Step 9: Run Ansible playbook

```bash
cd ansible
ansible-playbook -i inventory.ini site.yml
```

Expected behavior:

- First run performs setup and app deployment.
- Subsequent run should be mostly idempotent (`ok`/`skipping`, minimal or zero `changed`).

## Step 10: Verify application

- Open EC2 public IP in browser (HTTP).
- Confirm Nginx serves app.
- Confirm DB-backed pages/features work.

## Step 11: Re-run to confirm idempotency

```bash
cd ansible
ansible-playbook -i inventory.ini site.yml
```

A healthy repeat run should not keep changing unrelated tasks.

## 7. Day-2 Operations

### Update infrastructure

```bash
cd terraform/aws
terraform plan
terraform apply
```

If EC2 is replaced and IP changes, inventory is regenerated automatically.

### Redeploy app only

```bash
cd ansible
ansible-playbook -i inventory.ini site.yml --tags epicbook
```

Note: add role/task tags if you want granular deployments.

### Destroy infrastructure

```bash
cd terraform/aws
terraform destroy
```

Warning: this removes EC2/RDS and data unless protected by additional backup strategy.

## 8. Troubleshooting Guide

### Terraform: key pair file not found

Symptom:
- Error on `file(pathexpand(var.ssh_public_key_path))`

Fix:
- Ensure `.pub` file exists at that path on your local machine.

### SSH timeout

Symptom:
- `ssh: connect to host <ip> port 22: Connection timed out`

Fix:
- Use current IP from generated inventory.
- Verify SG inbound 22 and route/IGW configuration.

### MySQL access denied during seed

Symptom:
- `ERROR 1045 (28000): Access denied`

Fix:
- Align `db_user`, `db_name`, `db_password` between Terraform and Ansible.
- Confirm RDS endpoint in inventory matches active DB.

### Ansible changed every run

Symptom:
- Too many tasks report `changed` repeatedly.

Fix:
- Avoid unconditional `command`/`shell` without guards.
- Use state-aware modules/conditions and handlers.
